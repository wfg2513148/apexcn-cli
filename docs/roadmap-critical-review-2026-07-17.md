# Roadmap 360 度批判审查记录

## 审查范围

- 对象：`roadmap.json`、`issues.json`、测试反馈协议和版本执行规则。
- 独立审查任务：`019f6f14-0800-7dc3-b3a0-89448c11f23a`。
- 审查身份：独立 critic，不参与路线图实现。
- 目标：检查产品定位、阶段差异、验收可测性、跨仓库依赖、独立验证可信度和发版闭环。

## 初始结论

原路线图具备治理骨架，但不足以直接作为长期执行路线图。主要问题是：

1. 产品定位过宽，缺少主要用户、核心任务和明确排除对象。
2. 服务端、环境和外部依赖仅以字符串存在，不能阻止里程碑错误启动。
3. 量化指标缺少固定数据集、评分器、环境和逆向样本约束，存在指标失真风险。
4. 未开始能力容易被误写成活动 issue，造成虚假红灯或虚假绿灯。
5. MCP 共享核心层进入路线图过晚。
6. 产品阶段 `0.N` 与 SemVer release line 的映射不清楚。
7. 每轮固定复用 validator 线程会污染小白视角。

## 讨论与决策

### 已采纳

- 新增主要用户、次要用户、四类核心任务、排除用户和产品边界。
- 每个阶段增加用户结果、阶段非目标和结果指标。
- 采用结构化 dependency registry，区分 activation 和 completion 门禁。
- 规划性缺口进入 readiness risks；`issues.json` 只保存独立 validator 实际观察的问题。
- 所有验收条件绑定 measurement profile，固定 denominator、数据集、评分器、环境和证据要求。
- MCP readonly 与 shared core 前移到 `0.5`；可审计写工作流位于 `0.6`。
- `0.4` 聚焦个人工作台、能力协商和本地草稿；收藏转 collection 移至 `0.7`。
- `0.6` 回写验收要求后端/API 与真实浏览器视觉证据 100% 同时通过。
- 产品阶段 `0.N` 映射到 SemVer `0.N0.x`；里程碑进入使用 minor，目标模式日常迭代使用 patch。
- 每个里程碑在用户确认后启动一个独立 Codex 目标模式，直至 100% 验收和发版闭环。

### 调整后采纳

- Critic 建议允许部分迭代不发版。根据用户明确要求，不设置例外；改为精确定义目标模式，纯分析和未修改仓库的诊断不算目标模式迭代。
- Critic 建议拆分 `0.8`。当前保留单阶段，但限制为组织治理、兼容契约和生命周期资格化，不扩展远端 MCP 或无人确认写入。
- 不把 critic 发现的 metadata drift 直接写入 `issues.json`，而是作为验收条件和 readiness risk；只有独立 validator 实际复现后才能入 issue。

## Validator 线程新规则

固定的是测试项目，不是任务线程：

- 每轮验证都新建独立 Codex 任务线程。
- 每轮重置为小白身份，不复用旧线程上下文。
- 主会话依据当前里程碑、实际改动和风险动态生成 scope contract。
- 每轮同时执行适用固定基线套件和独立计分的动态里程碑/逆向探索套件。
- 旧线程只用于历史趋势、既有问题来源和关闭证据。

三项防漂移门禁：

1. Validator Thread Intake Gate：记录线程、项目、persona、被测版本、commit/tag、checksum 和允许读取材料。
2. Test Scope Contract Gate：记录里程碑、固定基线、动态场景、逆向目标、禁止项和证据格式。
3. Issue Admission Gate：每条 issue 绑定线程、任务、场景、首次失败、实际输出、期望结果和责任初判。

纵向指标绑定 artifact，而不是线程记忆。每轮记录 dataset、scorer、environment、CLI、server contract 和 thread 版本。

## 优化后阶段

| 阶段 | 差异化目标 |
|---|---|
| `0.2` | 可信赖的安装、认证、命令发现、契约、诊断和新手反馈 |
| `0.3` | 真实 ORDS 只读检索、来源、分页、RAG 质量与错误恢复 |
| `0.4` | Profile、个人状态、通知/规则/隐私、能力协商和本地草稿 |
| `0.5` | CLI/shared core 驱动的 readonly MCP 与真实客户端兼容 |
| `0.6` | Preview、approval、恢复、审计和双证据内容回写 |
| `0.7` | Collection、BM25、增量索引、收藏导入和只读自动化 |
| `0.8` | 组织策略、兼容契约、凭据与跨平台生命周期资格化 |
| `0.9` | 公开面冻结、恢复、供应链、安全审查和 soak 的 GA 候选资格 |

## 最终判断

优化后的路线图可以作为机器可执行的长期产品路线，但它仍只定义阶段目标和门禁。每个里程碑的实施计划必须由主会话在激活时根据最新代码、`roadmap.json`、`issues.json`、依赖证据和新 validator 任务即时生成。
